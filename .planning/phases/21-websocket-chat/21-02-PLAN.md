WS connection URL format: `ws[s]://host/ws?token=JWT&ticketId=UUID&role=resident|staff`

WS server sends JSON messages:
```typescript
{ type: "new_message", payload: TicketMessage }
// where TicketMessage = { id, ticketId, senderId, senderType, content, createdAt }
```

WS close codes:
- 1008: Policy violation (bad token)
- Normal close: server restart or client disconnect

  <action>
    Create `artifacts/scla/src/hooks/use-ticket-chat.ts` with:

    ```typescript
    export function useTicketChat(ticketId: string | undefined, token: string, role: "resident" | "staff", apiBase: string)
    ```

    Returns: `{ messages: TicketMessage[], isConnected: boolean, sendMessage: (content: string) => Promise<void> }`

    Implementation:
    1. **Initial fetch:** On mount (when ticketId truthy), fetch GET messages from REST to populate initial list. For resident: `GET {apiBase}/tickets/{ticketId}/messages`. For staff: `GET {apiBase}/admin/tickets/{ticketId}/messages`. Store in `useState<TicketMessage[]>`.

    2. **WebSocket connection:** After initial fetch, open WS:
       - Derive WS URL from apiBase: replace `http` with `ws`, append `/ws?token=${token}&ticketId=${ticketId}&role=${role}`
       - If apiBase is relative (starts with `/`), use `${location.protocol === 'https:' ? 'wss:' : 'ws:'}//${location.host}${apiBase.replace(/\/api$/, '')}/ws?...`
       - On `message` event: parse JSON, if `type === "new_message"`, append payload to messages state (deduplicate by id).
       - On `open`: set isConnected=true, disable polling.
       - On `close`/`error`: set isConnected=false, enable polling fallback.
       - Reconnect with exponential backoff: 1s, 2s, 4s, 8s, max 30s. Reset backoff on successful open.

    3. **Polling fallback:** When isConnected=false, use `setInterval` at 4000ms to re-fetch messages from REST. Clear interval when WS reconnects. This preserves existing behavior per RT-03.

    4. **sendMessage:** POST to REST endpoint (same as current). After successful POST, the WS broadcast will deliver the message back to all clients (including sender). The hook should optimistically append the message or rely on the WS broadcast.
       - For resident: POST `{apiBase}/tickets/{ticketId}/messages`
       - For staff: POST `{apiBase}/admin/tickets/{ticketId}/messages`

    5. **Cleanup:** On unmount or ticketId change, close WS, clear polling interval.

    Create `artifacts/admin/src/hooks/use-ticket-chat.ts` as a copy of the same hook. Both apps have the same hook logic — the `role` and `apiBase` params differentiate behavior. (Sharing via a package would require workspace setup; duplicating ~80 lines is pragmatic for two consumers.)

    The TicketMessage interface in both hooks:
    ```typescript
    interface TicketMessage {
      id: string;
      ticketId: string;
      senderId: string;
      senderType: "resident" | "staff";
      content: string;
      createdAt: string;
    }
    ```
