Create `.github/workflows/db-backup.yml`:

**Name:** `Daily DB Backup`

**Triggers:**
- `schedule`: cron `0 2 * * *` (daily at 2 AM UTC — early morning in Myanmar, MMT is UTC+6:30 so this is 8:30 AM)
- `workflow_dispatch` (manual trigger for on-demand backups)

**Job: `backup`**
- runs-on: `ubuntu-latest`
- Steps:
  1. Install PostgreSQL client: `sudo apt-get update && sudo apt-get install -y postgresql-client-16`
  2. Step "Create Backup" — Run pg_dump:
     ```bash
     TIMESTAMP=$(date +%Y%m%d_%H%M%S)
     BACKUP_FILE="scla_backup_${TIMESTAMP}.sql.gz"
     pg_dump "$DATABASE_URL" --no-owner --no-acl --clean --if-exists | gzip > "$BACKUP_FILE"
     echo "backup_file=$BACKUP_FILE" >> "$GITHUB_OUTPUT"
     echo "timestamp=$TIMESTAMP" >> "$GITHUB_OUTPUT"
     ```
     env: `DATABASE_URL: ${{ secrets.DATABASE_URL }}`
  3. Step "Verify Backup" — Check the backup file is non-empty and contains expected tables:
     ```bash
     BACKUP_FILE="${{ steps.dump.outputs.backup_file }}"
     SIZE=$(stat -f%z "$BACKUP_FILE" 2>/dev/null || stat --format=%s "$BACKUP_FILE")
     if [ "$SIZE" -lt 1000 ]; then
       echo "::error::Backup file too small ($SIZE bytes), likely failed"
       exit 1
     fi
     # Verify key tables are present in the dump
     zcat "$BACKUP_FILE" | grep -q "CREATE TABLE" || { echo "::error::No CREATE TABLE found in backup"; exit 1; }
     echo "Backup verified: $BACKUP_FILE ($SIZE bytes)"
     ```
     Give the dump step an `id: dump`.
  4. Step "Upload as Artifact" — Use `actions/upload-artifact@v4` to upload the backup file:
     ```yaml
     with:
       name: db-backup-${{ steps.dump.outputs.timestamp }}
       path: ${{ steps.dump.outputs.backup_file }}
       retention-days: 30
     ```
  5. Step "Cleanup old backups" — Add a comment noting that GitHub artifact retention handles cleanup automatically (30-day retention).

**Title:** Database Restore Procedure

**Sections:**

1. **Prerequisites**
   - Access to GitHub Actions artifacts (repo collaborator or higher)
   - PostgreSQL client tools (`pg_dump`, `pg_restore`, `psql`) version 16+
   - Target database connection string (DATABASE_URL)
   - Sufficient permissions to DROP/CREATE tables on the target database

2. **Locating Backups**
   - Navigate to GitHub repo > Actions > "Daily DB Backup" workflow
   - Each run has a `db-backup-YYYYMMDD_HHMMSS` artifact
   - Download the desired artifact (click the artifact name to download zip)
   - Or use CLI: `gh run download <run-id> -n db-backup-<timestamp>`
   - Backups are retained for 30 days

3. **Restore Steps**
   Step-by-step with exact commands:
   ```bash
   # 1. Download and extract backup
   unzip db-backup-YYYYMMDD_HHMMSS.zip
   
   # 2. Verify backup contents before restoring
   zcat scla_backup_YYYYMMDD_HHMMSS.sql.gz | head -50
   
   # 3. Restore to target database (WARNING: this replaces all data)
   zcat scla_backup_YYYYMMDD_HHMMSS.sql.gz | psql "$DATABASE_URL"
   
   # 4. Verify restore succeeded
   psql "$DATABASE_URL" -c "SELECT count(*) FROM users;"
   psql "$DATABASE_URL" -c "SELECT count(*) FROM invoices;"
   psql "$DATABASE_URL" -c "SELECT count(*) FROM tickets;"
   psql "$DATABASE_URL" -c "SELECT count(*) FROM bookings;"
   ```

4. **Verification Checklist**
   After restore, verify:
   - [ ] `users` table has expected row count
   - [ ] `invoices` table has expected row count
   - [ ] `tickets` table has expected row count
   - [ ] `bookings` table has expected row count
   - [ ] Application starts without errors: `pnpm run dev`
   - [ ] A resident can log in and see their dashboard
   - [ ] Admin portal shows correct KPI stats

5. **Restoring to a Different Environment**
   - Change DATABASE_URL to point to the target (staging, local)
   - After restore, run `pnpm --filter db push` to ensure schema is in sync
   - Update any environment-specific config (API URLs, webhook URLs)

6. **Troubleshooting**
   - "permission denied" — ensure the database user has CREATEDB or is the owner
   - "relation already exists" — the backup uses `--clean --if-exists` so this is expected to drop first; if issues persist, drop the database and recreate: `dropdb $DB_NAME && createdb $DB_NAME`
   - Backup file too small — check if DATABASE_URL secret is correctly set in GitHub; re-run the backup workflow manually
   - Partial restore — re-run the full restore; SQL dumps are transactional

7. **Emergency Contacts**
   - Database issues: Check Replit dashboard for database status
   - GitHub Actions: Check workflow run logs for error details
